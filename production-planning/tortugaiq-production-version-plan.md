## Overview

- The current demo app was built using Vite and localStorage.
- The production version will be a **full rebuild (reimplementation)** using a modern stack.
- The new app will be built with **Next.js**, including both frontend and backend logic.
- The app will use a **PostgreSQL database** (e.g., Neon).
- The goal is to evolve the demo into a **production-grade application** while keeping the system **simple, clear, and maintainable**.

---

## Architectural Philosophy

### Core Principles (in order)

1. **Correctness and data safety**
2. **Security and privacy**
3. **Maintainability and clarity**
4. **Simplicity over cleverness**
5. **Performance where it matters**
6. **Scalability without premature complexity**

---

### Simplicity Without Shallowness

We want a system that is:

- simple in architecture
- but **explicit, strongly typed, and professional in implementation**

We value:

- strong TypeScript usage
- explicit types and interfaces
- readable function signatures
- clear domain models
- predictable structure

We avoid:

- unnecessary abstractions
- architectural patterns with no clear payoff
- excessive indirection
- "AI-generated complexity"
- designing for imaginary scale

> Do not confuse simplicity with lack of structure.

---

## Stack

- **Framework:** Next.js (App Router)
- **Language:** TypeScript (strict usage)
- **Database:** PostgreSQL (Neon)
- **Styling:** Tailwind CSS
- **Other:** Other libraries or tools should be modern and widely used
- **Deployment:** Vercel
- **State Management (remote):** TanStack Query (if justified)
- **Global State:** Context API or Zustand (only if needed)
- **Logging:** Vercel runtime logs (`console.*`)
- **Authentication:** OAuth (Google, Facebook) + email/password

---

## UI Parity Requirement

The production app must preserve the behavior of the demo app.

Preserve:

- layout and visual structure
- interaction patterns
- keyboard behavior
- navigation flow
- feature behavior

Refactors are allowed, but **behavior regressions are not**.

---

## Backend Design

### Server Boundary

All sensitive logic must live on the server:

- authentication
- authorization
- database access
- ownership validation
- business rules

Never rely on client-side checks for security.

---

### Data Ownership Rule

Every user-owned record must:

- be associated with a `userId`
- enforce ownership at the server level

No user should ever access another user's data.

---

### Backend Organization

Structure backend code into:

- schema definitions
- migrations
- validation
- queries (reads)
- mutations (writes)
- auth logic
- shared business logic

Avoid scattering database logic across components.

---

## Database & Schema

### Initial Schema Philosophy

- keep schema small and simple
- model only current needs
- avoid premature normalization or fragmentation
- infer schema from the demo app

---

### Migration Discipline

- every schema change must have a migration file
- never manually modify production schema
- prefer additive changes first
- use expand → migrate → contract pattern
- commit migration files with related code
- test locally before production

---

### Data Safety

- user data is critical
- schema changes must preserve data
- destructive changes must be deliberate

---

### Backup Philosophy

For v1:

- backups are **manual and external**
- use periodic `pg_dump`
- store backups in multiple locations
- ensure dumps can be restored

Backups are **not part of normal app logic**, do not implement backup logic for v1.

---

## TypeScript Strategy

We rely heavily on TypeScript.

Goals:

- improve readability
- improve maintainability
- make data structures explicit

Rules:

- all main domain objects must have defined types
- function inputs and outputs and returns must be typed
- database models should map clearly to TypeScript types
- avoid unnecessary generic abstractions

Good TypeScript:

- clear
- explicit
- readable

Bad TypeScript:

- unnecessary abstractions

---

## Validation

All external input must be validated at runtime:

- request bodies
- query params
- form submissions
- auth inputs

TypeScript is not enough at runtime.

Validation must happen at server boundaries.

---

## Authentication

- OAuth (Google, Facebook)
- email/password flow
- password recovery
- secure session handling

Authentication must follow standard best practices.

UI updates are allowed but must match the existing design style.

---

## Error Handling

Use a standardized error handling approach:

- predictable error shapes
- clear separation of expected vs unexpected errors
- avoid leaking sensitive data
- handle errors both server-side and client-side

---

## Logging

### Strategy

Use Vercel runtime logs via:

- `console.error`
- `console.warn`
- `console.log` (sparingly)

### Rules

Log:

- operation names
- relevant IDs (userId, conceptId)
- error messages
- useful debugging context

Do NOT log:

- passwords
- tokens
- secrets
- full notes or sensitive content

Logs are for debugging, not data storage.

---

## Performance Philosophy

- prioritize perceived speed
- avoid unnecessary re-fetching
- keep interactions fast

Use TanStack Query if it clearly improves:

- caching
- performance
- user experience

Avoid complex caching strategies unless needed.

---

## Environment & Secrets

- use environment variables for all secrets
- never hardcode credentials
- separate environments (local / production)
- provide `.env.example`

---

## Testing Philosophy

Focus on meaningful tests:

- auth flows
- ownership rules
- core CRUD operations
- critical business logic

Avoid test overengineering.

---

## Definition of Done

A feature is complete only if:

- it works correctly
- it preserves UI behavior
- it respects auth and ownership
- it is properly typed
- it handles errors
- schema changes use migrations
- code is readable and maintainable
